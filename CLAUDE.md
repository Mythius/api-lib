# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`api-lib` is a **boilerplate/template** for spinning up new API backends — not a finished product. It provides a batteries-included Bun + Hono server with multi-provider authentication, session storage, and auto-generated CRUD routes over a Prisma schema. New projects are meant to start from this repo and build on top of it by editing `api.ts` and `prisma/schema.prisma`.

## Commands

- Install deps: `bun install`
- Run the server (dev/prod, no build step — Bun runs TS directly): `bun run index.ts` or `bun start`
- Generate the Prisma client after editing the schema: `bunx prisma generate` (output is written to `tools/generated/prisma`, matching the `generator client { output = ... }` in `prisma/schema.prisma`)
- Push schema changes to the database: **use `bunx prisma db push`, not `prisma migrate`** — this template doesn't use migration files, so always push the schema directly.
- Run via Docker (app + Postgres + Redis): `docker-compose up --build`
- Count lines of code per file: `bun run tools/count-lines.ts`
- Create a new local user (priv 1 = admin): call `POST /newuser` as an authenticated admin (see `tools/auth.ts`), or hand-edit `auth.json`
- Run an ad-hoc SQL file against `DATABASE_URL`: `bun run-sql.ts <file.sql>` (add `--json` for JSON output instead of a table). Use this to look at actual data — real rows, real shapes, real edge cases — before building a new feature or CRUD route against a model, rather than guessing from the schema alone.

There is no lint config, no test runner, and no build step configured in this repo — don't invent commands for these.

Runtime introspection: once the server is running, `GET /endpoints/html` (or `/endpoints/json`) lists every registered route. Check this first when you need to know what's actually mounted — it reflects the real registration order (including anything auto-generated by `exposePrismaCRUD`) rather than what you'd infer from reading `api.ts` alone. Try it against `http://localhost:<PORT>/endpoints/json` for local work; if the project has a known production URL, hit `<production-url>/endpoints/json` too so you're working from what's actually deployed, not just what's in the working tree.

## Architecture

**Entry point wiring (`index.ts`)** assembles the app in a specific, order-dependent sequence:
1. `expose(app)` from `tools/listEndpoints.ts` — adds the introspection routes.
2. Static file serving from `./public`.
3. `authPublic(app)` and `API.publicRoutes(app)` — routes reachable without a session.
4. `authMiddleware(app)` — a catch-all `app.use("*", ...)` that gates **everything registered after it**.
5. `authPrivate(app)` and `API.privateRoutes(app)` — routes that require a valid session.

Because Hono matches middleware/routes in registration order, anything added after step 4 requires auth by default. Public/private route separation in `api.ts` mirrors this — new endpoints go in `publicRoutes()` or `privateRoutes()` accordingly.

**`api.ts` is the project-specific layer.** It's the file a new project built from this template is expected to edit — everything under `tools/` is meant to stay generic/reusable. `api.ts` also wires an `onLogin` callback and calls `exposePrismaCRUD("api", app)` to mount the generated CRUD routes.

**Auth (`tools/auth.ts`)** supports four login paths that all converge on the same `Session` object and cookie-based `auth_token`:
- Local username/password (md5-hashed, stored in `auth.json` — a flat JSON file, not a database table).
- Google OAuth (redirect flow, or one-click credential verification via `/auth/google-oneclick`).
- Microsoft OAuth (redirect flow via Microsoft Graph).
- CAS (delegates to an external CAS server when `CAS_SERVER_URL`/`CAS_CLIENT_ID` are set — Google/Microsoft routes redirect to the CAS server instead of handling OAuth locally when CAS is configured).

Sessions are stored via a pluggable `SessionStore` (`MemoryStore` by default, `RedisStore` via `setSessionStore("redis")` in `index.ts`, backed by `REDIS_URL`). `auth.json` persists user records (password hash, privilege level, current session token) across restarts; it's gitignored and treated as a dev-only user store, not a production user database.

**Auto CRUD (`tools/prisma.ts` + `tools/createCRUD.ts`)**: `exposePrismaCRUD` reads every model off the generated `PrismaClient`, parses `prisma/schema.prisma` directly (via regex, in `parseSchema()`) to determine each model's primary key and foreign keys, and mounts a full REST CRUD surface per model at `/<prefix>/<model>` (list, get by id, paginated list, filter/query, create, update, delete), plus `/<prefix>/_schema` exposing the parsed schema. `createCRUD` accepts optional `checkPermissions` and `validateData` callbacks per-route for authorization and row-level filtering (`PermissionResult.rowLevelFilter` is merged into `where` clauses and enforced on writes) — this is the hook point for adding access control without touching the CRUD generator itself.

The Prisma datasource adapter (`pg` vs `mariadb`) is selected at runtime in `tools/prisma.ts` based on the scheme of `DATABASE_URL` (`postgres(ql)://`, `mysql://`/`mariadb://`).

**`run-sql.ts`** (repo root) runs an arbitrary SQL file against `DATABASE_URL` via Bun's built-in `SQL` client and prints the results as a table (or JSON with `--json`). It's the quickest way to inspect real data — reach for it before/while building a feature against a model, instead of relying solely on `prisma/schema.prisma` to infer shapes.

**Supporting tools (`tools/`)**:
- `fileUpload.ts` — multipart upload handler, writes to `./uploads` (gitignored).
- `mail.ts` — sends email via Gmail SMTP (nodemailer) or Linux `sendmail`, chosen by `EMAIL_TYPE` env var.
- `googleapi/google.ts` — Google Workspace API helpers (Drive, Apps Script) using a separate desktop-app OAuth flow (`tools/googleapi/credentials.json` + `token.json`, both gitignored) — unrelated to the end-user login flow in `tools/auth.ts`.

**`public/`** is a minimal demo frontend: `index.html` + `api.js` (fetch wrappers for the auth/CRUD endpoints) plus a small library of dependency-free vanilla-JS UI components (`combobox-input.js`, `date-range.js`, `file-drop.js`, `map-picker.js`, `rich-text.js`, `star-rating.js`, `tag-input.js`), demonstrated in `components/demo.html`. This is a reference/demo, not a framework — new frontends aren't required to use it.

**Deployment**: `Dockerfile` is a multi-stage Bun build (installs deps, runs `prisma generate` against a placeholder `DATABASE_URL` at build time, then copies into a slim runner image). `docker-compose.yml` runs the app alongside Postgres and Redis containers for local/prod-like use. `entrypoint.sh` is the container's start command.

## Environment variables

Copy `example-env` to `.env`. Key variables: `DATABASE_URL` (postgres/mysql/mariadb, required), `PORT`, `REDIS_URL` (if using the Redis session store), `CAS_SERVER_URL`/`CAS_CLIENT_ID`/`CAS_CALLBACK_URL`, `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET`/`GOOGLE_REDIRECT_URI`, `MS_CLIENT_ID`/`MS_CLIENT_SECRET`/`MS_REDIRECT_URI`, `GMAIL_USER`/`GMAIL_PASS`/`EMAIL_FROM`/`EMAIL_TYPE`. Routes for a given provider return a `503` with a "missing env vars" hint if that provider isn't configured, rather than failing silently.
